  C'EST QUOI XPVibing AU FOND ?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

XPVibing est une page web qui se fait passer pour un système
d'exploitation. C'est comme un décor de cinéma : de loin ça
ressemble à Windows XP, mais derrière c'est juste une page web
très bien costumée.

Le tout est construit avec trois ingrédients de base :
- HTML  → la structure (les boîtes, le texte, les boutons)
- CSS   → la déco (les couleurs, les tailles, les animations)
- JavaScript (React) → le comportement (ce qui se passe quand tu cliques)

L'outil utilisé pour assembler tout ça s'appelle Vite.
Imagine Vite comme un chef cuisinier qui prend tous les
ingrédients séparés et les cuit ensemble pour obtenir un seul
plat prêt à servir dans le navigateur.


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  REACT : L'ARCHITECTE DU PROJET
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

React est le framework (la bibliothèque) principal.
Analogie : React c'est comme des briques LEGO.
Chaque brique est un "composant" (un morceau d'interface
réutilisable). On assemble les briques pour former le tout.

Dans XPVibing, les briques principales sont :

  🖥️  Desktop       → le fond d'écran avec les icônes
  📌  Taskbar       → la barre en bas (bouton Start, horloge...)
  📋  StartMenu     → le menu qui s'ouvre avec le bouton Start
  🪟  Window        → le cadre gris de chaque fenêtre
  🔔  Notifications → la petite bulle en bas à droite
  💾  BootScreen    → l'écran de démarrage

Et les apps à l'intérieur des fenêtres :
  🎵 Music Player, 📝 Bloc-notes, 📁 Explorateur,
  🌐 Faux navigateur, 🐍 Snake, 💣 Démineur, 🕐 Horloge, etc.


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  LE "CERVEAU" DU PROJET : OSContext
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Il y a un fichier central qui sert de cerveau à tout le projet :
OSContext.jsx

Analogie : c'est le tableau de bord d'un avion.
Toutes les infos importantes y sont centralisées :
- Quelles fenêtres sont ouvertes ?
- Quel fond d'écran est actif ?
- Y a-t-il des notifications en attente ?
- Le démarrage est-il terminé ?

Quand tu cliques sur une icône pour ouvrir une app, tu envoies
un "message" au cerveau qui dit "ouvre cette fenêtre". Le cerveau
met à jour ses infos, et React rafraîchit automatiquement
l'écran pour refléter ce changement.

C'est comme un panneau d'affichage dans un grand bureau :
tout le monde regarde le même panneau, et si quelqu'un change
quelque chose dessus, tout le monde le voit immédiatement.


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  LES FENÊTRES : COMMENT ÇA MARCHE ?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Chaque fenêtre fonctionne comme une enveloppe et une lettre :
  - L'enveloppe = le composant Window (la barre de titre grise,
    les boutons minimiser/agrandir/fermer)
  - La lettre = l'app à l'intérieur (Bloc-notes, Snake, etc.)

On peut glisser les fenêtres parce qu'un petit morceau de code
(useDrag.js) écoute en permanence les mouvements de ta souris
quand tu cliques sur la barre de titre.

Analogie : c'est comme tenir un objet avec ta main. Tant que
tu maintiens le clic, le code "tient" la fenêtre et suit ton
curseur. Quand tu relâches, il "lâche" la fenêtre.

Pour savoir quelle fenêtre est au dessus des autres, chaque
fenêtre a un numéro invisible (z-index). La fenêtre avec le
plus grand numéro apparaît par dessus les autres.
Quand tu cliques sur une fenêtre, elle reçoit le numéro le
plus élevé du moment, et monte donc au premier plan.


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  L'ÉCRAN DE DÉMARRAGE : LE SPECTACLE D'INTRO
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Le boot screen est une mise en scène en 3 actes qui dure 3,5 secondes :

  Acte 1 (0 à 1,3s) : Le "BIOS"
    → Des lignes de texte apparaissent une par une, comme si
      une machine démarrait pour de vrai. C'est du pur décor.

  Acte 2 (1,3 à 3,1s) : La barre de chargement
    → Le logo XP Vibing apparaît avec une barre qui se remplit.
      Elle monte par petits bonds aléatoires pour faire naturel.

  Acte 3 (3,1 à 3,5s) : L'écran de bienvenue
    → L'icône utilisateur + "Bienvenue" + les petits points
      qui clignotent. Ensuite ça disparaît en fondu.

À la fin, le cerveau (OSContext) reçoit un signal "démarrage terminé"
et affiche le bureau.


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  LES SONS : D'OÙ VIENNENT-ILS ?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Il n'y a AUCUN fichier son dans le projet. Pas de .mp3, pas
de .wav. Les sons sont fabriqués en temps réel par le navigateur.

C'est possible grâce à la Web Audio API, une technologie
intégrée dans tous les navigateurs modernes.

Analogie : au lieu d'enregistrer quelqu'un qui joue du piano
(un fichier audio), on donne les partitions directement au
navigateur et il joue lui-même la note.

Les sons sont définis par des paramètres :
- La forme de l'onde (sine = son doux, square = son électronique,
  sawtooth = son agressif)
- La fréquence en Hertz (plus c'est élevé = plus c'est aigu)
- La durée en millisecondes

Voici les sons du projet :

  Ouvrir une fenêtre  → son doux qui monte (440 Hz → 880 Hz)
  Fermer une fenêtre  → son doux qui descend (660 Hz → 330 Hz)
  Clic de bouton      → bip électronique court (1200 Hz, 50ms)
  Erreur              → son rugueux grave (200 Hz, 300ms)
  Notification        → deux notes ensemble (880 Hz + 1100 Hz)
  Minimiser           → glissement vers le bas (550 Hz → 220 Hz)
  Démarrage           → petite mélodie de 4 notes (440, 550, 660, 880 Hz)
                        jouées l'une après l'autre avec 200ms d'écart

C'est la même technique utilisée par des jeux rétro 8-bit :
générer les sons mathématiquement plutôt que de les enregistrer.


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  LES ICÔNES : D'OÙ VIENNENT-ELLES ?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Là encore, aucun fichier image pour les icônes des apps.
Elles sont toutes des emojis Unicode.

Un emoji comme 🎵 ou 📁 n'est pas une image, c'est un caractère
de texte, comme la lettre A ou le chiffre 3, mais en coloré.
Tous les systèmes d'exploitation modernes incluent ces caractères.

Analogie : c'est comme si les icônes étaient des lettres dans
une police de caractères spéciale. On n'a rien à télécharger,
c'est déjà là dans l'ordinateur.

Les boutons de fenêtre (minimiser ─, agrandir □, fermer ✕)
sont eux dessinés en SVG, un format de dessin vectoriel que le
navigateur comprend nativement. C'est comme un schéma technique
décrit en coordonnées : "trace une ligne de ce point à ce point".

Le fond d'écran Bliss, lui, est une vraie image (ton fichier
bliss.jpeg copié dans le dossier public).

  LES FONDS D'ÉCRAN GÉNÉRÉS (Matrix, Espace, etc.)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Les autres fonds d'écran (Matrix rain, étoiles dans l'espace,
dégradés colorés) ne sont pas des images non plus.

Les dégradés sont décrits en CSS pur :
"fais un dégradé qui part du bleu ciel en haut et finit en
vert herbe en bas" → le navigateur le dessine lui-même.

La pluie Matrix et les étoiles sont des animations :
des dizaines de petits éléments HTML créés dynamiquement par
JavaScript, avec des positions et des vitesses aléatoires.

Analogie : c'est comme si au lieu d'une photo de pluie, on
prenait une boîte de cure-dents et qu'on les lançait un par
un depuis le haut de l'écran à des moments différents.


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  LA MÉMOIRE : CE QUI EST SAUVEGARDÉ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Le navigateur a un petit tiroir de rangement appelé localStorage.
XPVibing s'en sert pour retenir des choses entre les visites :

  - Le fond d'écran choisi
  - Le texte écrit dans le Bloc-notes
  - Le meilleur score au Snake
  - La position dans la playlist du Music Player

C'est comme des post-its collés sur ton navigateur.
Si tu vides le cache ou l'historique, les post-its disparaissent.


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  LE VISUALISEUR DU MUSIC PLAYER
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Le visualiseur de musique (les barres qui bougent) utilise
aussi la Web Audio API. Voici comment ça marche :

1. La musique joue dans un lecteur audio classique
2. La Web Audio API "écoute" la musique en même temps
3. Elle analyse les fréquences (les graves, les médiums, les aigus)
   60 fois par seconde
4. Elle transmet ces données à un Canvas (une toile de dessin)
5. Le Canvas efface et redessine les barres à chaque analyse

Analogie : imagine que tu branches un égaliseur graphique
(comme sur les vieilles chaînes hi-fi) à ta musique.
Les barres montent et descendent selon les fréquences.
Ici c'est la même chose, mais dessiné dans le navigateur
en temps réel.

  RÉSUMÉ EN UNE PHRASE PAR TECHNOLOGIE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  React       → les briques LEGO qui composent l'interface
  CSS         → la peinture et la déco sur les briques
  OSContext   → le cerveau qui garde en mémoire l'état du bureau
  Web Audio   → le synthétiseur qui fabrique les sons à la volée
  Emojis      → les icônes (caractères de texte colorés, pas des images)
  SVG         → les dessins vectoriels (boutons de fenêtre)
  localStorage → les post-its du navigateur pour sauvegarder
  Canvas      → la toile sur laquelle le visualiseur est dessiné
  Vite        → le cuisinier qui assemble tout pour le navigateur

  CE QUI N'EST PAS DANS LE PROJET
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ✗ Aucun fichier son (.mp3, .wav)
  ✗ Aucun fichier icône (.png, .ico, .svg d'app)
  ✗ Aucune base de données
  ✗ Aucun serveur (tout tourne dans ton navigateur)
  ✗ Aucune connexion internet nécessaire (sauf pour GitHub Pages)
  ✗ Aucune bibliothèque d'interface (pas de Material UI, Bootstrap...)

Tout est fait à la main, from scratch.
